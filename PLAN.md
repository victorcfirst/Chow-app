# Chow-app — แผนพัฒนา WebApp ครอบครัวเชาว์

> เวอร์ชันแผน: 1.0 · Stack: React + Vite + Supabase + Vercel (GitHub)
> เอกสารนี้คือภาพรวม/การตัดสินใจออกแบบ ส่วนกติกาการเขียนโค้ดอยู่ใน `CLAUDE.md`

---

## 1. เป้าหมาย

แอปกลางของบ้านเชาว์ ไว้จัดการเรื่องจุกจิกประจำวัน + นัดหมาย ให้คนในบ้าน
(หม่าม๊า / เฟิสท์ / มิ้นท์ / บอส / วิน) เปิดดูได้ที่เดียว ข้อมูลแชร์กันแบบ real-time

**Stack:** React + Vite, Supabase (DB + Storage + Realtime), Vercel, GitHub
**Auth:** PIN ร่วมทั้งบ้าน 4 ช่อง (ค่าเริ่มต้น `1234`) กั้นฝั่ง client — เปลี่ยนภายหลังได้
**ภาษา UI:** ไทยทั้งหมด · identifier ในโค้ด/DB เป็นอังกฤษ

## 2. หลักการออกแบบที่เป็นแกนกลาง (สำคัญ — ทำให้ทุกหน้าไม่กระจาย)

1. **`members` ตารางเดียวใช้ซ้ำทุกหน้า** — แต่ละคนมีสี + emoji เก็บที่เดียว แล้วเอาไปใช้
   เป็น tag ในโน้ต / ชื่อผู้สั่งอาหาร / เจ้าของประกัน / ผู้เกี่ยวข้องในนัด
   → component `<MemberPicker>` (multi-select toggle แบบหน้าสิทธิ Nira-app) เขียนครั้งเดียวใช้ทุกที่
   เพิ่มคนในอนาคต (แฟน/ลูก) แก้จุดเดียว

2. **Reminder engine กลาง (`useReminders` + `reminders.js`)** — รวม logic นับถอยหลัง +
   สถานะสี (เขียว/เหลือง/แดง/เลยกำหนด) ไว้ที่เดียว ดึงจากหลายแหล่งมารวม:
   นัดหมาย, สิทธิประโยชน์, งวดประกันถัดไป, วันคุ้มครองหมด, ต่อภาษีรถ, พ.ร.บ., ประกันรถ, เช็กระยะ
   → **ไม่ duplicate ข้อมูล** ใช้วิธี "คำนวณรวม" ฝั่ง client ไม่ยัด due-date ซ้ำลง events

3. **Dashboard หน้าแรก (ใต้ Welcome)** — การ์ดสรุป "ที่ต้องรู้วันนี้": นัดใกล้ถึง,
   อะไรครบกำหนดใน 30 วัน, ออเดอร์อาหารวันนี้, โน้ตปักหมุด/ล่าสุด

### หมายเหตุการตัดสินใจ — ปฏิทิน "สิทธิประโยชน์" (Nira-app)
ในบรีฟแรกมีทั้ง *ปฏิทินสิทธิประโยชน์* (แบบ Nira-app) และ *ปฏิทินนัดหมาย* แยกกัน
**ผมรวมเป็นปฏิทินเดียว** โดยใช้ field `category` (`appointment` = นัดหมาย, `benefit` = สิทธิประโยชน์)
หน้า Calendar มีปุ่ม toggle กรองหมวด → ได้ทั้งสองอย่างในระบบเดียว ไม่ต้องเขียนซ้ำ
**ถ้าเฟิสท์อยากให้แยกเป็นคนละหน้าจริง ๆ บอกได้** เดี๋ยวปรับ schema/หน้าให้

## 5. Data model (สรุป — รายละเอียดเต็มใน `sql/schema.sql`)

```
app_settings(key, value)                       -- เก็บ pin
members(id, name, color, emoji, sort_order, active)
family_photos(id, storage_path, caption, sort_order)
notes(id, content, created_by, pinned, done) ── note_members(note_id, member_id)
restaurants(id, name, phone, note)
  └ menu_items(id, restaurant_id, name, price)
  └ orders(id, restaurant_id, order_date, status)
       └ order_items(id, order_id, member_id, menu_name, qty, note)
events(id, title, category, event_date, event_time, location) ── event_members(...)
insurance_policies(id, owner_id, ...ฟิลด์ประกัน..., tax_deductible, doc_path)
vehicles(id, category, ...ฟิลด์รถ..., reg_book_image_path, insurance_image_path)
```

**model ออเดอร์อาหาร:** 1 order = 1 ร้าน ต่อ 1 วัน (เปิดให้ทุกคนมาเติม `order_items` ของตัวเอง)
ใครตื่นก่อนเห็นสรุปรวม → โทรสั่ง → กดเปลี่ยน status เป็น `called`
`menu_name` เก็บ snapshot ชื่อเมนูกันเมนูถูกลบ. ข้าวเหนียว ฯลฯ = order_item อีกบรรทัด

## 6. ฟิลด์หน้าประกัน + การสรุป

**ฟิลด์:** เจ้าของ, ประเภท (ชีวิต/สุขภาพ/อุบัติเหตุ/โรคร้ายแรง), บริษัท, ชื่อกรมธรรม์, เลขกรมธรรม์,
นายหน้า + เบอร์, ผู้รับผลประโยชน์, ทุนประกัน, เบี้ย/งวด, ความถี่ชำระ, ชำระกี่ปี,
วันเริ่ม, คุ้มครองถึง, งวดถัดไป, งวดที่ X/Y, ยอดลดหย่อน, สถานะ, ไฟล์กรมธรรม์

**ฟิลเตอร์:** นายหน้า · เจ้าของ · ลดหย่อนภาษีได้ (มี/ไม่มี)
**การ์ดสรุปด้านบน:** เบี้ยรวมต่อปี (แยกตามคน) · ยอดลดหย่อนรวมต่อคน

> **เบี้ยต่อปี** = `premium × ความถี่/ปี` (year=1, semi=2, quarter=4, month=12)
> **ลดหย่อน:** แอปแค่ "บวกยอดที่กรอกเอง" ให้ — ไม่ใช่คำแนะนำภาษี
> เพดานลดหย่อน (ประกันชีวิต/สุขภาพ/ของพ่อแม่) ให้เช็กกฎสรรพากรปีนั้นหรือนักบัญชีอีกที

## 7. ฟิลด์หน้ารถ

ข้อมูลรถ (ยี่ห้อ/รุ่น/ปี/ทะเบียน/จังหวัด) · ต่อภาษีครบกำหนด · พ.ร.บ. หมด ·
ประกันภาคสมัครใจ (ชั้น/บริษัท/เลขกรมธรรม์/วันหมด) · เลขไมล์ล่าสุด · เช็กระยะถัดไป ·
รูปเล่มทะเบียน + รูปประกัน (กดดูเพิ่มเติม) · แยกหมวด ครอบครัว/บริษัท

## 8. Security & Privacy ⚠️ (อ่านก่อนตัดสินใจ)

หน้าประกัน/รถมีข้อมูลอ่อนไหว (เลขกรมธรรม์, เลขทะเบียน, รูปเล่มทะเบียน)

**ทำใน MVP:**
- **รูปภาพ → bucket `documents` แบบ private + signed URL** เท่านั้น ไม่เปิด public, ไม่เอาไว้ใน URL params
- แกลเลอรีครอบครัว → bucket `gallery` (public-read ได้ ความอ่อนไหวต่ำ — สลับเป็น private ได้ถ้าต้องการ)

**ข้อจำกัดที่ต้องรู้ (ขอพูดตรง ๆ):**
PIN เป็นแค่ประตูนุ่มฝั่ง client. RLS แบบ MVP เปิดให้ anon อ่าน/เขียนได้ (เหมือน CK-app)
ดังนั้นใครมี URL + anon key (ติดมากับ bundle) ก็เข้าถึง DB ได้ตรง ๆ — PIN กั้นไม่ได้จริง

**ถ้าต้องการป้องกันข้อมูลประกัน/รถจริงจัง** → เฟสถัดไปย้ายไป **Supabase Auth (login รายคน)**
แล้วเขียน RLS ผูกกับ `auth.uid()` อันนี้ค่อยทำตอนพร้อม ไม่บล็อก MVP

## 9. แจ้งเตือน / PWA

- **MVP:** แจ้งเตือนในแอป (badge/card บน dashboard + สีสถานะจาก reminder engine) — พอแล้ว
- **PWA installable:** เปิดด้วย `vite-plugin-pwa` (manifest + ติดตั้งลงจอโฮมได้) — ทำเลยใน MVP เพราะง่าย
- **Push notification:** ❌ ยังไม่ทำใน MVP. Push จริงต้องมี service worker + Web Push (VAPID) +
  ตาราง `push_subscriptions` + cron ส่งแจ้งเตือน (Supabase Edge Function) — งานปานกลาง
  วางโครงให้เพิ่มภายหลังได้โดยไม่ต้องรื้อ (ดู CLAUDE.md หัวข้อ PWA)
